# 🚀 기술 면접 대비: 활용 기술 심화 학습 가이드

이 문서는 이력서의 [기술활용경험] 항목에 기재한 4가지 핵심 기술(언어, 프레임워크, DB, AI)에 대해 면접관의 꼬리 질문이 들어왔을 때, 흔들림 없이 '나만의 언어'로 답변할 수 있도록 돕는 학습 자료입니다.

## 1. 개발 언어: JavaScript (및 Python)

### 📌 이력서 기재 내용
*   JavaScript(중): 한글 자소 분해 알고리즘 및 픽셀 단위 이미지 제어(Canvas) 구현.
*   Python(중): 웹 크롤링 및 데이터 전처리 스크립트 작성.

### 💡 핵심 이해 포인트 (왜 이 언어인가?)
*   **JavaScript (웹의 지배자)**: 브라우저 위에서 사용자의 클릭, 드래그에 즉각 반응하고 화면을 다시 그리기 위해 필수적입니다.
*   **Canvas API 제어**: `<canvas>` 태그 위에서 `getImageData` 함수를 사용해 글자가 그려진 픽셀(RGBA 데이터)을 추출하고, 여백을 계산해 자동으로 크롭(자르기)하는 수학적 계산을 수행했습니다.

### 🎯 예상 꼬리 질문 & 모범 답변
*   **Q. 한글 자소 분해 알고리즘을 기존 라이브러리(Hangul.js 등)를 쓰지 않고 JS로 직접 구현한 이유는 무엇인가요?**
    *   **A.** 단순히 초/중/종성을 분리하는 것이 목표가 아니었기 때문입니다. '아래아(ㆍ)' 같은 고어 처리 규칙과, 분해된 자소들이 이루는 형태(예: 받침 없는 수직형)를 18가지 마스터 유형으로 라벨링하는 커스텀 로직이 필요했습니다. 이를 위해 유니코드의 규칙성(0xAC00부터 시작)을 이용한 나눗셈, 나머지 연산을 JS 코드로 직접 작성했습니다.
*   **Q. 파이썬 크롤링 파이프라인 경험이 있다고 했는데, 주로 어떤 라이브러리를 썼나요?**
    *   **A.** (본인의 이전 경험에 맞게 답변: 예) BeautifulSoup이나 Selenium을 활용해 DOM 구조를 파싱하고 데이터를 수집한 뒤, Pandas를 이용해 결측치를 전처리했습니다.

## 2. 프레임워크: React & Vite

### 📌 이력서 기재 내용
*   React(중): `useReducer`를 활용한 중앙 상태 관리 로직 적용.
*   Vite(하): 브라우저 네이티브 모듈(ESM) 기반 고속 빌드 환경 구성.

### 💡 핵심 이해 포인트 (왜 React인가?)
*   **상태(State) 기반 렌더링**: X좌표, Y좌표 등의 '상태'가 변하면 화면이 자동으로 갱신됩니다.
*   **`useReducer`의 역할**: 수량 하나 올리고 내리는 건 `useState`로 충분하지만, 누리한글처럼 수십 개의 슬라이더 수치(X, Y, W, H, 두께 등)가 얽혀있을 때는 계산 로직이 꼬입니다. 그래서 '이 값을 이렇게 바꿔라!'라는 명령서(Action)만 보내면, 중앙 통제소(Reducer)가 모든 수치 연산을 깔끔하게 알아서 처리하도록 아키텍처를 설계했습니다.

### 🎯 예상 꼬리 질문 & 모범 답변
*   **Q. `useState` 대신 `useReducer`를 사용해서 얻은 이점이 정확히 무엇인가요?**
    *   **A.** 상태 업데이트 로직을 컴포넌트(UI)에서 분리할 수 있었다는 점입니다. 수십 개의 슬라이더를 움직일 때마다 복잡한 연산이 필요한데, 이를 UI 컴포넌트 안에 두면 코드가 비대해집니다. Reducer라는 중앙 통제소에서 연산을 전담하게 하여 코드의 가독성과 유지보수성을 크게 높였습니다.
*   **Q. CRA(Create React App) 대신 Vite를 쓴 이유는 무엇인가요?**
    *   **A.** 조판 프로그램의 특성상 1픽셀의 수정이 화면에 어떻게 반영되는지 즉각적으로 확인하는 HMR(Hot Module Replacement) 속도가 생명입니다. 모든 코드를 미리 다 묶어버리는 기존 Webpack(CRA)과 달리, 브라우저의 기본 모듈(ESM)을 활용해 필요한 부분만 빠르게 교체해 주는 Vite가 개발 생산성 측면에서 압도적으로 유리했습니다.

## 3. 데이터베이스: Firebase Firestore (NoSQL)

### 📌 이력서 기재 내용
*   Firestore(중): NoSQL 구조 설계 및 `onSnapshot` 리스너를 활용한 실시간 동기화 구축.

### 💡 핵심 이해 포인트 (왜 NoSQL인가?)
*   **Document 기반 저장**: 관계형 DB(MySQL)처럼 딱딱한 표(Table) 형식이 아니라, 자유로운 JSON 객체 덩어리(문서)를 그대로 저장할 수 있습니다. 
*   **`onSnapshot` (웹소켓 기반 실시간 통신)**: 데이터베이스의 값이 바뀌면 새로고침 없이 즉각 브라우저로 알림을 쏴주는 Firebase의 강력한 기능입니다.

### 🎯 예상 꼬리 질문 & 모범 답변
*   **Q. RDBMS(MySQL 등) 대신 NoSQL(Firestore)을 선택한 이유는 무엇인가요?**
    *   **A.** 두 가지 이유입니다. 첫째, 팀원들이 저장하는 자소별 좌표 데이터는 고정된 스키마(표)보다는 유연한 JSON 형태(객체 트리)로 저장하고 불러오는 것이 프론트엔드(React) 상태와 바로 매핑하기 좋았습니다. 둘째, 피그마(Figma)처럼 내가 조정한 비율이 다른 사람의 화면에도 즉시 반영되는 '실시간 협업(onSnapshot)' 기능을 가장 쉽고 빠르게 구현할 수 있는 최적의 툴이었기 때문입니다.

## 4. AI/Data: GPT/LLM & 프롬프트 엔지니어링

### 📌 이력서 기재 내용
*   GPT/LLM(중): 프롬프트 체이닝을 활용해 알고리즘 및 시스템 아키텍처 구현 주도.

### 💡 핵심 이해 포인트 (비전공자의 무기)
*   면접관의 우려: "그냥 AI한테 코드 짜달라고 해서 복붙한 거 아니야?"
*   방어 논리: "AI는 큰 그림(아키텍처)을 보지 못합니다. 전체 파이프라인을 기획하고, AI의 환각을 통제하며 단계별로 모듈을 조립한 것은 '나'입니다."

### 🎯 예상 꼬리 질문 & 모범 답변
*   **Q. '프롬프트 체이닝(Prompt Chaining)' 기법을 썼다고 하셨는데, 구체적으로 어떻게 적용했나요?**
    *   **A.** AI에게 "한글 타이포그래피 엔진을 만들어줘"라고 한 번에 지시하면 무조건 환각(오류)이 발생하거나 엉뚱한 코드를 줍니다. 따라서 저는 전체 아키텍처를 세 단계로 쪼갰습니다. 첫 번째 프롬프트에서는 '유니코드 분해 로직'만, 두 번째는 'Canvas 렌더링 로직'만, 세 번째는 'Firebase 연결 로직'만 별도로 요구했습니다. 각 단계에서 검증된 코드만 다음 단계의 입력값(Context)으로 제공하여, AI의 환각을 완벽히 통제하고 제가 원하는 엔터프라이즈급 구조를 완성했습니다.

## 5. 비전: 머신러닝 파이프라인 도입 (자소서 내용 방어)

### 📌 자소서 기재 내용 방어
*   **내용:** "수집된 좌표 데이터를 전처리하여 시스템이 최적의 배치를 스스로 예측하는 머신러닝 파이프라인의 도입..."

### 💡 핵심 이해 포인트 (어떻게 현재 프로그램과 연동되는가?)
1. **데이터 수집 (현재 구현 완료):** 사용자가 슬라이더를 움직여 최적화한 초/중/종성의 X, Y, 크기, 굵기 좌표 데이터가 현재 Firebase Firestore(`nuri_glyphs` 컬렉션)에 **'라벨링된 정답 데이터'**로서 실시간으로 적재되고 있습니다.
2. **데이터 추출 및 전처리 (Next Step):** Python 스크립트를 작성하여 Firestore의 JSON 데이터를 추출합니다. 유니코드 자소와 마스터 유형(예: 'V-1-0')을 범주형 데이터(Categorical Feature)로 인코딩하고, 좌표값들을 0~1 사이로 정규화(Normalization)합니다.
3. **모델 학습 및 예측 API (Target):** Random Forest나 가벼운 신경망 모델을 학습시켜, '새로운 글자'의 특징이 입력되었을 때 최적의 좌표값을 예측(Regression)합니다. 학습된 모델은 REST API(예: FastAPI 또는 Firebase Cloud Functions)로 배포합니다.
4. **프로그램 연동 (Integration):** Nuri_Typo 앱(`App.jsx`)에서 사용자가 아직 작업하지 않은 완전히 새로운 글자를 타이핑하면, 기존의 하드코딩된 `DEFAULT_MASTER_TYPES`를 사용하는 대신 이 ML API를 호출합니다. 모델이 예측한 좌표값을 초기 상태(Base State)로 렌더링하여, 디자이너의 수작업 시작점을 획기적으로 줄여줍니다.

### 🎯 예상 꼬리 질문 & 모범 답변
*   **Q. 머신러닝 파이프라인을 도입하겠다고 했는데, 구체적으로 어떤 데이터를 학습시킬 계획인가요?**
    *   **A.** 현재 제 프로그램은 디자이너가 직접 조정한 자소별 X, Y, Scale 등의 수치 데이터가 실시간으로 Firestore DB에 저장되는 아키텍처를 갖고 있습니다. 즉, 사람이 직접 시각적으로 검수하여 완성한 **'정답 좌표(Target)'가 자동으로 라벨링**되어 쌓이고 있습니다. 이 데이터를 파이썬으로 추출하여, 초/중/종성의 형태학적 분류 키(예: V-1-0)를 입력값(X)으로, 그에 따른 최적의 렌더링 좌표 수치들을 예측값(Y)으로 하는 회귀(Regression) 모델을 학습시킬 계획입니다.
*   **Q. 학습된 모델은 현재 구현된 프론트엔드와 기술적으로 어떻게 연동되나요?**
    *   **A.** 학습된 파이썬 모델을 FastAPI로 래핑하여 추론용 REST API 서버를 구축할 것입니다. React 프론트엔드(`App.jsx`)에서는 사용자가 한글을 입력했을 때 로컬이나 DB 캐시에 없는 글자라면, 이 API를 호출해 AI가 예측한 최적의 좌표값을 받아옵니다. 그리고 이 값을 `useReducer`의 초기 상태(Payload)로 주입하여 렌더링함으로써, 데이터 기반의 자동 조판을 구현할 것입니다.
