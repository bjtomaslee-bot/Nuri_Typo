# 누리 타이포그래피 (Nuri_Typo) 면접 대비 기술 정리 노트

이 문서는 자소서에 작성된 기술적 내용과 프로그램의 실제 구조를 면접관에게 자신 있게 설명하기 위한 커닝페이퍼(Cheat Sheet)입니다. 달달 외우기보다는 '내가 왜 이 기술을 썼는지' 흐름을 이해하는 데 집중해 주세요.

---

## 1. 핵심 로직: 한글 엔진 (HangulEngine)
면접관이 가장 궁금해할 "한글을 어떻게 쪼갰는가?"에 대한 답입니다.

*   **유니코드 수학 연산**: 한글은 유니코드에서 `가(0xAC00)`부터 시작하여 규칙적으로 배열되어 있습니다. 입력된 글자에서 기본값(0xAC00)을 빼고, 나눗셈과 나머지 연산(Modulo, `%`)을 통해 초성, 중성, 종성의 인덱스 번호를 역산해 냅니다.
*   **고어(아래아 등) 처리**: 표준 유니코드 연산 후, '아래아(ㆍ)' 같은 특수 모음이 아래에 붙는지, 오른쪽에 붙는지 판단하여 변수를 추가 할당합니다.
*   **마스터 유형(Master Types) 분류**: 분해된 자소들의 조합 형태(예: 모음이 아래에만 있는지, 받침이 2개인지 등)를 분석하여 `V-1-0`(수직형-받침없음), `C-2-2`(복합형-쌍받침) 등 18가지 이상의 고유한 구조적 유형으로 자동 라벨링을 합니다.

## 2. 프론트엔드 상태 관리: React `useReducer`
"왜 useState만 쓰지 않았나요?"라는 질문에 대한 답입니다.

*   **도입 이유**: 이 프로그램은 X좌표, Y좌표, 장평(W), 높이(H), 굵기 등 조절해야 할 슬라이더(상태)가 수십 개입니다. 이를 `useState`로 개별 관리하면 코드가 스파게티처럼 꼬입니다.
*   **해결**: `useReducer`를 도입하여 '상태(State)'와 상태를 변경하는 '행동(Action, 예: UPDATE, NUDGE)'을 분리했습니다. 슬라이더를 움직이거나 키보드 방향키를 누르면 중앙 통제소(Reducer)에 신호만 보내고, 그곳에서 모든 수치 연산을 깔끔하게 처리하도록 아키텍처를 짰습니다.

## 3. 그래픽 렌더링: Canvas API & SVG
화면에 보여주고 이미지로 뽑아내는 기술입니다.

*   **Canvas API (픽셀 제어)**: 상태가 바뀔 때마다 브라우저의 `<canvas>`에 글자를 실시간으로 그립니다. 특히 `getImageData` 함수를 사용해 캔버스의 모든 픽셀을 검사하여, 글자가 있는 부분만 정확히 계산해 내는 **여백 자동 크롭(Bounding Box)** 알고리즘을 직접 구현했습니다.
*   **SVG 내보내기**: 픽셀 깨짐이 없는 벡터 이미지를 위해, 현재 좌표와 크기 상태 값을 SVG HTML 태그 문자열로 조립하여(`.toDataURL` 사용) 파일로 다운로드할 수 있게 만들었습니다.

## 4. 백엔드/클라우드: Firebase Firestore
자소서의 '데이터 파이프라인'과 '협업' 부분입니다.

*   **NoSQL 데이터베이스**: 실시간 동기화가 가장 강력한 Firebase Firestore를 사용했습니다.
*   **저장 구조**: 
    1. `nuri_glyphs` (개별 글자 데이터): 팀원이 특정 글자(예: '갋')의 비율을 예쁘게 깎아 저장하면, 이 좌표값 객체(JSON)가 그대로 DB에 저장됩니다.
    2. `nuri_master_types` (유형 데이터): 특정 유형(예: 받침 없는 수직형) 전체의 기본 비율을 저장합니다.
*   **실시간 반응형**: `onSnapshot` 리스너를 달아서, 다른 팀원이 클라우드에 값을 덮어쓰면 내 화면의 슬라이더와 글자 모양도 새로고침 없이 실시간으로 즉시 바뀌도록 구성했습니다.

## 5. 빌드 도구: Vite
*   **도입 이유**: 기존의 Create React App(CRA)은 Webpack 기반이라 무겁고 느립니다. Vite는 최신 브라우저 모듈(ES Modules)을 사용하여 서버 구동 속도와 핫 리로드(HMR) 속도가 압도적으로 빨라, 실시간 UI 렌더링 확인이 필수인 본 프로젝트에 도입했습니다.

## 🎯 면접 예상 질문 & 방어 가이드 (Q&A)

**Q1. 본인이 프롬프트 엔지니어링으로 기획을 주도했다고 하셨는데, AI가 짠 코드에서 본인은 어떤 역할을 했나요?**
A: AI는 단위 모듈(예: 캔버스 크롭 함수)의 코드를 짜는 데는 훌륭하지만, 전체 아키텍처의 연결성은 모릅니다. 저는 전체 파이프라인(한글 분해 -> 상태 관리 -> 렌더링 -> DB 저장)을 설계하고, AI에게 단계별로 명확한 컨텍스트(프롬프트 체이닝)를 제공하여 시스템을 조립하는 '아키텍트' 역할을 수행했습니다.

**Q2. 한글 제자 원리 알고리즘은 기존 라이브러리를 쓰지 않고 왜 직접 구현했나요?**
A: 기존 라이브러리(예: Hangul.js)는 단순히 초/중/종성을 분리하는 데 그칩니다. 하지만 누리 타이포그래피는 '아래아' 같은 고어 요소를 포함해야 하고, 형태적 특징에 따라 18가지 마스터 유형으로 라벨링하여 초기 좌표값을 다르게 주어야 했기 때문에, 목적에 맞는 커스텀 엔진을 직접 설계할 수밖에 없었습니다.

**Q3. 자소서에서 '수동 매핑과 하드코딩의 한계'를 느꼈다고 하셨는데, 정확히 어떤 문제였나요?**
A: 현재 코드를 보면 18개의 마스터 유형(`DEFAULT_MASTER_TYPES`)에 대해 X, Y, 장평 등의 초기값이 수치로 직접 입력(하드코딩)되어 있습니다. 사용자가 늘어나고 예외적인 문자 조합 데이터가 많아질수록, 이 최적의 비율을 사람이 일일이 계산해서 코드를 수정하는 것은 불가능합니다. 

**Q4. (꼬리질문) 그럼 SKALA에서 배운 기술로 이 문제를 어떻게 해결할 건가요?**
A: Firestore에 누적된 팀원들의 수백, 수천 개의 '예쁘게 조판된 좌표 데이터'를 추출(Extract)하고 전처리(Transform)하는 파이프라인을 구축하겠습니다. 그리고 이 데이터를 바탕으로, 새로운 글자가 입력되었을 때 최적의 X, Y 좌표값을 예측해주는 경량화 머신러닝 모델을 만들고, 이를 MLOps 기반으로 자동 배포하여 하드코딩 없이 시스템이 스스로 진화하게 만들고 싶습니다.
